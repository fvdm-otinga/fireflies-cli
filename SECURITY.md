# Security Policy

## Supported Versions

| Version | Security patches |
|---------|-----------------|
| 1.0.x   | Yes             |
| < 1.0   | No              |

## Reporting a Vulnerability

**Do not file a public GitHub issue.** Instead, email
[francois@otinga.io](mailto:francois@otinga.io) with:

- A description of the vulnerability and its potential impact
- Steps to reproduce or a proof-of-concept
- Any suggested remediation you have in mind

You will receive an acknowledgement within **7 days**. We will keep you informed
as the issue is investigated and resolved. See [Disclosure Timeline](#disclosure-timeline)
for what to expect next.

## Threat Model

### In Scope

| Surface | Controls |
|---------|----------|
| **Webhook server** (`webhooks serve`) | HMAC-SHA256 signature verification using `hmac.Equal` (constant-time); request body capped at 1 MiB; `recover()` guard prevents panics from crashing the server |
| **GraphQL client** | TLS 1.2+ enforced by Go's default `crypto/tls`; response body capped before JSON decode |
| **Realtime WebSocket** (`realtime tail`) | `wss://` scheme enforced; `SetReadLimit` prevents unbounded memory growth from large frames |
| **File upload** (`meetings upload`) | Upload URL validated against `https://storage.googleapis.com`; rejects non-HTTPS or non-GCS targets (SSRF guard) |
| **Config / credentials** | Config file written with mode `0600`, directory `0700`; API key not echoed on login prompt; masked in `config list` output |

### Out of Scope

- **Supply-chain attacks on dependencies** — mitigated by `govulncheck` in CI and a
  CycloneDX SBOM attached to every release; we do not accept reports for transitive
  CVEs already tracked upstream.
- **Compromise of the user's machine** — an attacker with local access can read any
  file the user can; no additional hardening is provided beyond standard file permissions.
- **Compromise of `api.fireflies.ai`** — the upstream Fireflies.ai service is outside
  our control.

## Known Limitations

### Webhook Replay Protection

Fireflies does not currently include a signed timestamp in webhook payloads, so the
server cannot enforce a replay-window (e.g. reject requests older than 5 minutes).
An attacker who captures a valid webhook request can re-submit it indefinitely.

**Mitigation:** Operators who need replay protection should run the webhook server
behind a reverse proxy that deduplicates incoming requests by request-body hash
(e.g. using a short-lived Redis set). We will add native replay protection if and
when Fireflies adds a signed timestamp to their payloads.

### Custom GraphQL Endpoint Override

The `endpoint` config field allows self-hosted or staging deployments to point the
CLI at a non-default GraphQL server. The API key is sent to whichever endpoint is
configured. The CLI prints a warning to `stderr` when a non-default endpoint is
active, but it performs no additional validation of the target host.

**Mitigation:** Operators are responsible for verifying the endpoint URL. Do not
set a custom endpoint to a value you do not control.

### No Certificate Pinning

TLS certificate validation relies on the system trust store. A compromised or
rogue CA in the system store could perform a MITM attack.

**Mitigation:** Users on untrusted networks (public Wi-Fi, corporate intercept
proxies) should use a VPN or otherwise ensure the integrity of their trust store.

## Release Verification

Every release includes:

- **SHA256 checksums** (`fireflies_<version>_checksums.txt`) — verify with:
  ```sh
  shasum -a 256 -c fireflies_<version>_checksums.txt
  ```
- **CycloneDX SBOM** (`fireflies_<version>_sbom.json`) — machine-readable inventory
  of all Go module dependencies, signed alongside the release artifacts.

Checksums and the SBOM are generated by [GoReleaser](https://goreleaser.com) and
[Syft](https://github.com/anchore/syft) in the release CI pipeline.

## Disclosure Timeline

We follow **coordinated disclosure**:

1. Reporter contacts us privately.
2. We acknowledge within 7 days.
3. We develop and test a fix (target: 30 days for critical, 90 days for moderate/low).
4. We release a patched version and publish a security advisory on GitHub.
5. Reporters are credited in the release notes unless they request anonymity.

We ask that reporters refrain from public disclosure until a patch is available or
90 days have elapsed from the initial report, whichever comes first.
